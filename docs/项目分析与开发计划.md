# FlowNode 项目分析与开发计划

> 本文档记录 FlowNode 项目的整体分析、同类项目对比、技术栈优缺点，以及后续分阶段开发计划，供长期参考。

---

## 一、项目分析

**FlowNode 是一个基于 C# WinForms + .NET Framework 4.7.2 的可视化节点 / 流程图编辑器**，思路类似 Unreal Blueprint 或 Node-RED 的简化版。零 NuGet 依赖，纯 .NET BCL 实现。

### 1.1 技术栈一览

| 维度 | 内容 |
|------|------|
| 应用类型 | Windows 桌面应用 —— 可视化节点 / 流程图编辑器 |
| 语言 | C# |
| UI 框架 | Windows Forms（`System.Windows.Forms`） |
| 目标框架 | .NET Framework 4.7.2 |
| 输出类型 | `WinExe` |
| 构建工具 | MSBuild / Visual Studio 2019+（`.sln` + 旧式 `.csproj`） |
| 第三方库 | 无 NuGet 依赖，仅引用 BCL |
| 渲染 | GDI+ 自绘（`System.Drawing.Drawing2D` 贝塞尔曲线） |
| 序列化 | `System.Xml.Serialization.XmlSerializer`（XML） |
| 反射 | 节点自动发现与注册（`NodeFactory`） |

### 1.3 架构分层（Model–View–Command）

```
node/            ← 数据模型 + 执行引擎（与 UI 解耦）
  NodeBase / Pin / Connector / NodeManager / NodeFactory
  SystemNode/    ← Sequence / Branch / Loop / Function 等内置节点

app/view/        ← 编辑器 UI 与交互
  NodeEditor.cs  ← 画布核心：网格、贝塞尔连线、缩放平移、HitTest、状态机调度
  NodeView.cs    ← 单个节点视图：绘制、引脚布局、内嵌控件
  states/        ← 状态机：Idle / Connecting / DraggingNode / Selecting / Panning
  control/       ← 节点内自定义 GDI+ 控件

app/command/     ← 命令模式（Undo / Redo）
app/serialization/ ← XML 序列化持久化
```

#### 核心类关系

- `NodeEditor` 用 `Dictionary<INode, NodeView>` 维护**数据 ↔ 视图**双向映射。
- `NodeManager` 持有图数据（节点、连接），并承担**执行引擎**（栈式调度）。
- `NodeView`（抽象）→ `DefaultNodeView` / `TestNodeView`，每个 `NodeView` 绑定一个 `NodeBase`。
- `NodeViewFactory` 负责"节点类型 → 视图类型"映射，启动时注册。
- `CommandManager` 维护撤销栈 / 重做栈，支持命令组。

#### 内置系统节点

| 节点 | 路径 | 功能 |
|------|------|------|
| `SequenceNode` | `/system/Sequence` | 顺序执行 |
| `BranchNode` | `/system/Branch` | 条件分支 True/False |
| `LoopNode` | `/system/Loop` | 循环（LoopBody / Completed） |
| `FunctionNode` | 动态 | 反射调用静态方法 |
| `GetObjectNode` / `SetObjectNode` | 动态 | 读写全局变量 |

#### 状态机交互

| 状态 | 触发 | 行为 |
|------|------|------|
| `IdleState` | 默认 | 点引脚→连接；点节点→拖拽；空白→框选；中键→平移；右键连线→删除 |
| `ConnectingState` | 拖拽引脚 | 绘制预览贝塞尔线，松手创建连接 |
| `DraggingNodeState` | 拖拽节点 | 移动选中节点，松手记录 Move 命令 |
| `SelectingState` | 空白左键拖拽 | 框选多个节点 |
| `PanningState` | 中键拖拽 | 平移画布 |

#### 执行引擎流程

1. `DemoForm` 点击 Execute → `NodeEditor.ExecuteFlow()` → `NodeManager.run()`
2. 找入口节点（无输入 Execute 连接的节点）
3. 栈式调度：`node.run(manager)` → `resolve()`（拉取上游数据）→ `excute()`（业务逻辑）→ `pushNextConnectNode()`（推进下游）

### 1.4 设计亮点

- **双轨 Pin 系统**：Execute（控制流）+ Data（数据流），与 Blueprint 一致。
- **状态机驱动交互**：鼠标交互拆成独立 State 类，`NodeEditor` 只做调度，扩展性好。
- **命令模式做撤销重做**：增删节点 / 连线 / 移动都封装成 Command，支持命令组。
- **反射注册节点**：`[SystemNode]` / `[Node]` / `[Function]` 特性 + `NodeFactory` 自动发现，新增节点成本低。
- **数据与视图解耦**：`node/` 逻辑层不依赖 UI，便于未来迁移 / 测试。

### 1.5 成熟度与遗留问题

核心框架已成型，但有若干"半成品"与代码隐患：

- `NodeSearchMenu`（节点搜索弹窗）字典为空，未完成。
- `PropertyPanel`、`VariableListControl` 已写但未接入主窗口（被注释）。
- `Form1.cs` 是早期原型（自带一套同名 `NodeView`/`Node`/`Connection`），**未纳入 .csproj 编译**，属死代码。
- `new PinType()` 写法脆弱：恰好等于 `PinType.Execute`（枚举默认值 0），一旦枚举顺序变动即出错，应显式写 `PinType.Execute`。
- `Program.cs` 中 `AllocConsole()` 始终开调试控制台，发布版应去掉。
- README 仅两行，无使用 / 扩展说明；无单元测试、无 CI。

## 二、同类项目对比

### 同类完整产品 / 框架
- **Node-RED**（JS，可视化流程编排，IoT / 自动化领域事实标准）
- **Unreal Engine Blueprint**（本项目的设计原型）
- **Blender Geometry / Shader Nodes**、**Unity Visual Scripting (Bolt)**
- **ComfyUI**（AI 绘画节点工作流，近两年爆火）
- **n8n**（工作流自动化，Node-RED 的商业化竞品）

### 可直接复用的 .NET 节点编辑器库（重点参考）
- **Nodify**（WPF，最活跃的 .NET 节点编辑器控件库，社区质量高）
- **NodeNetwork**（WPF + ReactiveUI）
- **ST.Library.UI / NodeGraph**（WinForms 老牌节点控件）
- **NodeEditorWinforms**（WinForms，与本项目最接近）

### 其他生态的画布库
- Web：**React Flow / Rete.js / LiteGraph.js / Drawflow**
- C++：**imgui-node-editor**（Dear ImGui 生态）
- Qt：**NodeEditor (QtNodes)**

**结论**：该领域非常成熟，本项目本质是"自研 WinForms 版的 Blueprint"。作为学习 / 自研可控引擎有价值；若目标是快速产品化，Nodify(WPF) 或 Web 系（React Flow）能省掉大量底层渲染 / 交互工作。

## 三、技术栈优缺点（WinForms + .NET Framework 4.7.2 + GDI+ 自绘）

### 优点
- **上手快、所见即所得**：WinForms 设计器友好，桌面应用开发效率高。
- **零外部依赖**：纯 BCL，编译即跑，无供应链 / 版本问题。
- **GDI+ 自绘可控**：节点外观、连线、命中检测完全自掌控。
- **生态稳定、文档海量**：老技术，问题几乎都能搜到答案。
- **部署简单**：Windows 上单 exe 即可运行。

### 缺点（决定能否长期演进）
- **平台锁定**：.NET Framework 仅限 Windows，无法跨平台。
- **框架已停止演进**：4.8 是最后大版本，微软主推 .NET 5/6/7/8+，新特性与性能优化都在新平台。
- **GDI+ 性能瓶颈**：CPU 软件渲染，无硬件加速；节点上千、频繁缩放重绘时会卡。
- **高分屏（DPI）适配麻烦**：自绘需手动处理缩放，4K 屏易糊 / 错位。
- **无现代数据绑定 / MVVM**：状态同步靠手写事件，UI 复杂后维护成本上升。
- **UI 现代化天花板低**：动画、阴影、圆角、主题等做起来费劲，观感偏传统。

**一句话**：适合"快速做出能用的 Windows 工具 / 原型"，不适合"长期演进、跨平台、大规模高性能"的产品方向。

## 四、分阶段开发计划

### 阶段一：止血与基线（优先级最高）
1. **清理死代码**：删除 / 隔离 `Form1.cs` 这套旧 `NodeView`，避免概念混淆。
2. **修隐患**：`new PinType()` → 显式 `PinType.Execute`。
3. **去掉发布版控制台**：`AllocConsole()` 用 `#if DEBUG` 包裹。
4. **补 README**：怎么运行、怎么加自定义节点、XML 格式说明。

### 阶段二：补齐核心功能闭环
5. **接好节点搜索菜单**（`NodeSearchMenu`）：从 `NodeFactory` 反射结果填充，支持右键空白处搜索建节点。
6. **接入属性面板**（`PropertyPanel`）和**变量面板**（`VariableListControl`）到 `DemoForm`。
7. **完善文件菜单**：新建 / 打开 / 保存 / 另存，配合 XML 序列化。
8. **复制 / 粘贴、多选批量操作、对齐**等常规编辑能力。

### 阶段三：执行引擎健壮性
9. **图校验**：环检测、孤立节点、类型不匹配连线检查。（已完成：入口节点检查 + 执行流环检测 + 孤立节点/未连接数据输入引脚告警）
10. **执行可视化**：运行时高亮当前节点、断点、单步、日志面板。（已完成：当前节点高亮 + 日志面板 + 断点 + 单步/继续/停止）
11. **错误处理**：节点 `run()` 异常时不崩溃，能定位到具体节点。（已完成：异常被捕获记入日志，并把高亮以红色停在出错节点上）

### 阶段四：质量与性能（持续）
12. **渲染优化**：脏区重绘 / 视口裁剪（只画可见节点）。
13. **DPI 适配**：高分屏测试。
14. **单元测试**：先覆盖 `node/`（纯逻辑层），再覆盖 Command 的 Undo/Redo。
15. **CI**：GitHub Actions 做 build + test。

### 战略决策（投入阶段二前应明确长期方向）
- **坚持 Windows 桌面、长期维护** → 建议先把 `node/` 逻辑层迁到 .NET 8，UI 层逐步从 WinForms 迁到 **WPF + Nodify**。
- **跨平台 / 多人协作 / Web 化** → 逻辑层迁 .NET 8 做后端，前端换 **React Flow**，C# 当执行引擎。
- **学习 / 内部小工具** → 保持现状，按阶段一~三推进即可。

## 附录：执行记录

- **阶段一**（已完成）：清理 `Form1.*` 死代码、修复 `new PinType()` → `PinType.Execute`、`AllocConsole` 加 `#if DEBUG`、完善 README。
- **阶段二**（进行中）：
  - 接通节点搜索菜单 `NodeSearchMenu`：改为从 `NodeFactory.GetNodePath()` 拉取全部已注册节点路径，用 `NodeFactory.CreateNode(path)` 创建；区分屏幕坐标（菜单位置）与画布坐标（节点位置）；加入关键字过滤、回车/双击创建、失焦/Esc 关闭等弹窗交互。
  - 在 `IdleState` 右键空白处弹出搜索菜单，在鼠标所在画布位置建节点。
  - 接入属性面板 `PropertyPanel`：在 `DemoForm` 右侧停靠一个 `PropertyGrid`，选中单个节点时显示其属性，多选/空选时清空。
  - 在 `NodeEditor` 新增 `SelectionChanged` 事件，并在 `ClearSelection`/`AddToSelection`/`RemoveFromSelection` 及删除节点处触发；`DemoForm` 订阅该事件刷新属性面板（替代早期 `MouseUp`/`KeyUp` + `BeginInvoke` 的临时方案，更可靠且覆盖框选/删除/程序化操作）。
  - 变量面板未接入 `VariableListControl`：它与已接入的 `DataViewControl` 功能重复且实现粗糙，沿用 `DataViewControl` 管理全局变量。
  - 完善文件菜单：`New`（确认后清空图）/`Open`/`Save`/`Save As...`。新增 `NodeEditor.NewGraph()` 清空图（节点、连接、变量、选中、撤销历史），`DataViewControl.RefreshList()` 供新建/打开后刷新变量列表；`DemoForm` 跟踪 `currentFilePath` 实现 Save 直存、Save As 弹框，并在标题栏显示当前文件名。
  - 复制/粘贴与全选：`Ctrl+A` 全选，`Ctrl+C` 复制选中节点及其内部连线到内存剪贴板，`Ctrl+V` 粘贴（整体偏移、自动选中、作为单次可撤销操作）。通过 `NodePath` + `NodeFactory.CreateNode` 重建节点、按引脚名重建连线；无 `NodePath` 的节点（如变量节点）暂不支持复制。批量删除已由原有 `Delete` 覆盖。
- **阶段三**（进行中）：
  - 图校验：`NodeManager.Validate()` 检查"图为空 / 无入口节点 / 执行流存在环"，`run()` 执行前先校验，未通过则抛错；`NodeEditor.ExecuteFlow()` 校验失败时在日志列出原因并中止。`run()` 增加步数上限（100 万）防止意外死循环卡死界面。环检测只针对执行引脚连接，循环节点靠运行时重压栈实现、不会被误报。
  - 执行可视化：`NodeManager` 新增 `NodeExecuting`/`Log` 事件，`run()` 每步触发；`NodeEditor` 执行时高亮当前节点（绿色发光边框）并按 `ExecutionStepDelayMs`（默认 200ms）单步停顿，执行结束/异常后清除高亮。
  - 日志面板：`DemoForm` 底部停靠只读多行 `TextBox`，订阅 `NodeEditor.ExecutionLog` 实时显示执行/校验/错误日志。
  - 单步执行与断点：把 `NodeManager` 执行引擎从"一次跑完"重构为可分步驱动（`BeginRun`/`Step`/`StopRun`/`PeekNext`/`IsRunning`/`HasMoreSteps`，`run()` 改为委托这些原语）。`NodeEditor` 提供 Run/Continue、Step、Stop，并在命中断点处暂停；`F9` 对选中的单个节点切换断点，断点节点左上角画红点。工具栏新增 Run/Continue、Step、Stop 三个按钮。执行节奏（高亮停顿）由 `RunLoop` 控制，单步则无停顿。
  - 收尾：`NodeManager.ValidateWarnings()` 输出告警（孤立节点、未连接且无默认值的数据输入引脚），执行前以 `[警告]` 记入日志但不阻断；节点执行抛异常时把高亮以**红色**停在出错节点上便于定位（`executionError` 标记，下次执行/停止/新建时清除）；`StopRun(silent)` 支持出错时静默复位；`NewGraph` 一并清空断点与执行状态。
